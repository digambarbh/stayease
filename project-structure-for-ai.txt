HOTEL BOOKING WEBSITE - PROJECT STRUCTURE AND RELATION GUIDE
=============================================================

Purpose of this file:
This file explains the current project structure, model relations, route flow, and intended booking/payment logic so another AI assistant can understand the app quickly and answer questions correctly.


1. PROJECT OVERVIEW
-------------------

This is a Node.js, Express, EJS, MongoDB, and Mongoose hotel/stay booking website.

Main features currently present:
- User registration
- User login/logout using JWT stored in cookies
- Host-only stay creation
- Stay listing and stay detail page
- Booking form page
- Booking model with payment fields
- Booking route is incomplete and still being developed

Main technology:
- Express 5
- EJS
- ejs-mate layouts
- MongoDB/Mongoose
- JWT authentication
- Cookie based login
- bcrypt password hashing
- method-override for PATCH/DELETE forms

Important package note:
- Razorpay is NOT currently installed in package.json.
- If payment is implemented with Razorpay, install it with:
  npm install razorpay

3. SERVER ENTRY POINT
---------------------

File:
server.js

Responsibilities:
- Loads environment variables using dotenv.
- Creates Express app.
- Connects to MongoDB using ./db/db.
- Sets EJS and ejs-mate.
- Enables express.urlencoded for form data.
- Enables cookie-parser.
- Enables method-override.
- Adds global middleware to read JWT token from cookie and set:
  res.locals.currentUser
  res.locals.isLoggedIn
- Mounts route files.
- Defines universal error handler.
- Starts server using process.env.SERVER_PORT.

Mounted routes:

app.use("/user", userRoute)
app.use("/stays", stayRoute)
app.use("/bookings", bookingRoute)

This means:
- routes/user.js controls /user/*
- routes/stay.js controls /stays/*
- routes/booking.js controls /bookings/*


4. AUTHENTICATION FLOW
----------------------

Authentication uses JWT in a cookie named "token".

Login/register creates token:
- Payload contains:
  id: user._id
  role: user.role
- Token expires in 7 days.
- Token is stored in cookie.

server.js global middleware:
- Reads req.cookies.token.
- If token is missing, user is treated as logged out.
- Checks Blacklist collection to reject logged-out tokens.
- Verifies token.
- Loads user from DB.
- Sets res.locals.currentUser and res.locals.isLoggedIn.

middleware/auth.js:
- Used to protect private routes.
- Reads token from req.cookies.token.
- Checks blacklist.
- Verifies JWT.
- Sets req.user = payload.id.
- Redirects to /user/login when user is not logged in.

Important:
Routes that need logged-in user id should use checkAuth middleware and then use req.user.
Do not manually decode JWT again inside every route unless necessary.


5. MIDDLEWARE
-------------

middleware/auth.js

Purpose:
- Protects routes that require login.
- Sets req.user to logged-in user id.

Usage:
router.get("/some-private-route", checkAuth, handler)


middleware/isHost.js

Purpose:
- Allows only users with role "host".
- It expects req.user to already exist.

Correct usage:
router.get("/new", checkAuth, isHost, handler)

Important:
isHost must come after checkAuth because it needs req.user.


6. MODELS AND RELATIONS
-----------------------

There are four models:
- User
- Stay
- Booking
- Blacklist


6.1 USER MODEL
--------------

File:
model/user.js

Fields:
- username: String, required, unique
- email: String, required, unique
- password: String, required
- phone: String, required
- avatar: String, default null
- role: String enum ['user', 'host', 'admin'], default 'user'
- wishlist: array of Stay ObjectIds
- stay: array of Stay ObjectIds
- bookings: array of Booking ObjectIds

Relations:

User -> wishlist -> many Stays
User -> stay -> many hosted Stays
User -> bookings -> many Bookings

Note:
The field name "stay" in User is an array. It may be clearer as "stays" or "hostedStays" later, but currently it is named "stay".


6.2 STAY MODEL
--------------

File:
model/stay.js

Fields:
- name: String, required
- description: String, required
- price: Number, required
- capacity: Number, required
- location: String, required
- images: array of Strings
- type: String enum ['apartment', 'villa', 'cabin', 'cottage', 'hotel']
- isAvailable: Boolean, default true
- host: User ObjectId
- bookings: array of Booking ObjectIds

Relations:

Stay -> host -> one User
Stay -> bookings -> many Bookings

Important:
The capacity field is named "capacity".
The booking form currently uses stay.maxGuests, which is wrong.
It should use stay.capacity.


6.3 BOOKING MODEL
-----------------

File:
model/booking.js

Current fields:
- checkIn: Date, required
- checkOut: Date, required
- guests: Number, required
- totalPrice: Number, required
- status: enum ['pending', 'approved', 'rejected', 'cancelled'], default 'pending'
- payment.status: enum ['pending', 'paid', 'refunded'], default 'pending'
- payment.razorpayOrderId: String
- payment.razorpayPaymentId: String
- user: User ObjectId, required
- stay: Stay ObjectId, required
- host: User ObjectId, required

Relations:

Booking -> user -> one User who made the booking
Booking -> stay -> one Stay that was booked
Booking -> host -> one User who owns the stay

Recommended missing fields:
- nights: Number, required
- specialRequests: String

Reason:
views/booking/booking.ejs sends nights and specialRequests.
Currently the Booking schema does not store these.


6.4 BLACKLIST MODEL
-------------------

File:
model/blacklist.js

Purpose:
- Stores JWT tokens after logout.
- Prevents a logged-out token from being reused.

11. RECOMMENDED BOOKING CREATION LOGIC
--------------------------------------

Use checkAuth:

router.post("/:id", checkAuth, async (req, res, next) => {
  try {
    const stay = await Stay.findById(req.params.id)

    if (!stay) {
      return res.send("Stay not found")
    }

    const checkIn = new Date(req.body.checkIn)
    const checkOut = new Date(req.body.checkOut)
    const nights = Math.ceil((checkOut - checkIn) / (1000 * 60 * 60 * 24))

    if (nights <= 0) {
      return res.send("Check-out must be after check-in")
    }

    if (Number(req.body.guests) > stay.capacity) {
      return res.send("Guest count is greater than stay capacity")
    }

    const totalPrice = nights * stay.price

    const booking = new Booking({
      checkIn,
      checkOut,
      guests: req.body.guests,
      nights,
      totalPrice,
      specialRequests: req.body.specialRequests,
      user: req.user,
      stay: stay._id,
      host: stay.host
    })

    await booking.save()

    stay.bookings.push(booking._id)
    await stay.save()

    await User.findByIdAndUpdate(req.user, {
      $push: { bookings: booking._id }
    })

    res.redirect("/bookings/my")
  } catch (err) {
    next(err)
  }
})


12. PAYMENT IMPLEMENTATION PLAN
-------------------------------

Payment provider planned:
Razorpay

Required install:
npm install razorpay

Required .env keys:
RAZORPAY_KEY_ID=your_key_id
RAZORPAY_KEY_SECRET=your_key_secret

Recommended payment flow:

1. User opens stay page.
2. User clicks book.
3. GET /bookings/new/:stayId renders booking form.
4. User submits booking form.
5. POST /bookings/:stayId validates booking data.
6. Server recalculates nights and totalPrice.
7. Server creates Booking with payment.status = "pending".
8. Server creates Razorpay order.
9. Server saves Razorpay order id in booking.payment.razorpayOrderId.
10. Server renders payment page with booking, order, and Razorpay key id.
11. User pays using Razorpay Checkout.
12. Razorpay frontend handler sends these fields to backend:
    razorpay_order_id
    razorpay_payment_id
    razorpay_signature
13. POST /bookings/:bookingId/verify verifies signature.
14. If valid, booking.payment.status = "paid".
15. If invalid, booking.payment.status = "failed" or keep pending.
16. Redirect user to /bookings/my.

Important:
Do not mark booking as paid until signature verification succeeds.


13. RAZORPAY ORDER CREATION SHAPE
---------------------------------

Example:

const order = await razorpay.orders.create({
  amount: totalPrice * 100,
  currency: "INR",
  receipt: `booking_${booking._id}`
})

Why amount * 100:
Razorpay expects amount in paise, not rupees.


14. RAZORPAY SIGNATURE VERIFICATION SHAPE
-----------------------------------------

Use crypto:

const body = razorpay_order_id + "|" + razorpay_payment_id

const expectedSignature = crypto
  .createHmac("sha256", process.env.RAZORPAY_KEY_SECRET)
  .update(body)
  .digest("hex")

if (expectedSignature !== razorpay_signature) {
  // payment verification failed
}

If successful:

booking.payment.status = "paid"
booking.payment.razorpayPaymentId = razorpay_payment_id
await booking.save()

16. IMPORTANT IMPLEMENTATION RULES
----------------------------------

Use req.user from checkAuth:

Correct:
router.post("/:id", checkAuth, async (req, res) => {
  console.log(req.user)
})

Avoid this in booking route:
const token = req.cookies.token
const payload = jwt.verify(token, process.env.JWT_SECRET)

Create Booking with one object:

Correct:
new Booking({
  checkIn,
  checkOut,
  guests,
  nights,
  totalPrice,
  user: req.user,
  stay: stay._id,
  host: stay.host
})

Wrong:
new Booking(req.body, { user: req.user }, { host: stay.host })

Wrong:
new Booking({ req.body })

18. SHORT SUMMARY FOR AI ASSISTANT
----------------------------------

This project is an Express/Mongoose hotel booking app.
Users authenticate through JWT cookies.
Hosts can create stays.
Normal users can book stays.
A Booking connects:
- user: who booked
- stay: what was booked
- host: who owns the stay

Booking payment should be:
- Create booking as pending.
- Create Razorpay order.
- Store order id in booking.payment.razorpayOrderId.
- Verify payment signature after checkout.
- Mark booking.payment.status as paid only after verification.

The current booking route is incomplete and should be rewritten around checkAuth, Mongoose relations, server-side price calculation, and Razorpay verification.
